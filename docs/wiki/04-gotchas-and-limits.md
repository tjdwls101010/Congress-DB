# 04 — 함정과 경계

이 DB는 국회 원천 데이터를 **그대로(raw fidelity)** 보존합니다. 결측과 표기 불일치가 그대로 남아 있어서, 모르고 쓰면 **에러 없이 조용히 틀린 답**이 나옵니다. 쿼리가 실패하면 알아차리지만, 분모가 잘못된 찬성률은 그럴듯한 숫자로 보고서에 들어갑니다.

모든 함정의 정량 분포는 컬럼 `COMMENT`에 박혀 있습니다 — `\d+ <table>`이 가장 정확하고 항상 최신입니다(COMMENT는 DB와 함께 움직이지만 이 문서는 낡을 수 있습니다). 아래는 실제 감사에서 사람과 AI 에이전트가 가장 자주 걸린 것들입니다.

## A. 집계를 조용히 틀리게 만드는 것

**1. `'불참'`을 분모에 넣기** — `result_vote_mod`의 `'불참'`은 빠진 행이 아니라 **저장된 값**이고, 482,714행 중 121,017행(약 25%)입니다. 출석률·찬성률의 분모는 `count(*) FILTER (WHERE result_vote_mod <> '불참')`으로 잡아야 합니다. `count(*)`로 나누면 수 %p 낮게 나옵니다.

**2. `votes`에서 가결률 계산하기 (생존편향)** — `votes`에는 본회의 표결까지 간 의안만 있고 그 대부분이 가결된 것들입니다. 여기서 "가결률"을 내면 100%에 가까운 무의미한 값이 나옵니다. 가결 여부는 `bills.proc_result`로 봅니다.

**3. `count(*)`를 의안 수로 읽기** — `votes`의 row 단위는 `bill_id × mona_cd`입니다. 의안 수는 `count(DISTINCT bill_id)`.

**4. 발의주체 커버리지** — **가결 법안의 약 64%는 대표발의자 행이 아예 없습니다**(위원장 대안 약 768건, 정부제출안 약 196건). `bill_lead_proposers`만 JOIN해 "정당별 입법 성과"를 세면 절반 이상이 조용히 사라집니다. 위원장 대안의 실제 발의자는 `bill_lineage`로 원안을 되짚어야 나옵니다.

**5. 발의자 정당 NULL 20명** — 떠난 의원(`is_incumbent = false`)은 `members.poly_nm`이 NULL이라 정당 GROUP BY에서 조용한 NULL 버킷으로 빠집니다(공동발의 216,537행의 약 5.2%). 시점 정당이 필요하면 `votes.poly_nm_at_vote`로 보정하되, `members.poly_nm`을 덮어쓰지는 마세요.

## B. 조용히 0행을 만드는 것

**6. 공포를 `bill_id`로 join** — `bill_final_outcomes`의 키는 **`bill_no`** 입니다. 반드시 `o.bill_no = b.bill_no`.

**7. `vote_date`를 DATE와 직접 비교** — `vote_date`는 TIMESTAMPTZ라 DATE 컬럼과 등치 비교하면 **조용히 0행**입니다. 그리고 `vote_date::date`는 세션이 GMT라 늦은 시각 표결을 하루 어긋나게 뽑습니다. 일 단위 비교·조인은 전부 생성컬럼 **`vote_date_kst`** 로 하세요.

**8. `proc_result = '가결'`** — 그런 값은 존재하지 않습니다. 통과는 `IN ('원안가결','수정가결')`입니다.

**9. `prom_law_nm`으로 법제처 매칭** — 66건이 NULL이고 이름 있는 것의 약 절반이 공백 제거형이라 exact-match가 실패합니다. bridge 키는 **`prom_no`**(공포 건 100% 채움)를 쓰고, 이름 매칭이 불가피하면 `prom_law_nm_norm`을 쓰되 **상대측에도 같은 정규화를 적용**하세요.

**10. `(대안)·(정부)` 법안의 NULL 타임라인** — `committee_dt`·`cmt_proc_dt`·`law_proc_dt`가 구조적으로 전부 NULL입니다. 이 날짜로 `INNER JOIN`하면 **법이 될 가능성이 가장 높은 대안·정부안이 통째로 빠집니다.** `LEFT JOIN`을 쓰세요.

## C. 잘못된 결론으로 이끄는 것

**11. `proc_result IS NULL`을 부결로 읽기** — NULL(13,745건, 약 71%)은 **미처리**입니다. 실제 `'부결'`은 2건뿐입니다.

**12. `law_proc_dt`를 공포일로 쓰기** — 이건 법사위 처리일이고 항상 공포일보다 이릅니다. 공포일은 `bill_final_outcomes.promulgation_dt`입니다.

**13. "가결인데 공포 없음"을 곧장 계류·거부권으로 세기** — 결의안·감사요구안 등 **애초에 공포 대상이 아닌 의안 171건**이 섞여 있습니다. 생성컬럼 `bills.is_law_bill`로 먼저 거르세요.

**14. 재의결 표결을 찾으려 하기** — 대통령 거부권 후 재의결된 법안은 본회의 표결이 두 번이지만, **원천 API가 원표결만 제공해 `votes`엔 하나만 있습니다**(26건 확인). 재의결 여부는 `bill_final_outcomes.plenary_dt > bills.proc_dt`로 식별하고, 재의결 표는 회의록·외부 자료로 넘어가야 합니다.

**15. 동명이인** — 박지원 2명처럼 같은 이름이 존재합니다. `hg_nm`으로 JOIN하면 서로 다른 두 사람의 데이터가 합쳐집니다. 식별은 **반드시 `mona_cd`**.

**16. `bill_name`을 식별자나 입장 표시로 읽기** — 이름이 심하게 중복되고(고유 이름은 전체의 약 1/5), **같은 이름의 법안이 정반대 방향일 수 있습니다**(예: 노조법). 이름은 식별자가 아닙니다.

**17. `bill_lineage` 0행 ≠ 미흡수** — 4,204행 중 500행은 canonical 대안으로 해소되지 않아 `alternative_bill_id IS NULL`입니다. `bills.cmt_proc_result`도 함께 확인하세요.

## D. 신선도 — 없는 게 아니라 아직 안 들어온 것

도메인마다 적재 시점이 다릅니다. **"미공포다"·"계류 중이다"·"최근 표결이 없다"를 단정하기 전에 반드시 확인하세요.**

```sql
SELECT * FROM data_freshness ORDER BY domain;
```

2026-07-19 시점 실측으로 `bills`의 최신 사실은 2026-07-16, 공포 이력은 2026-07-07, **표결은 2026-06-18**이었습니다. 즉 "최근 한 달간 표결이 없다"는 결론은 데이터가 아직 안 들어온 것일 수 있습니다. 산출물에는 항상 기준일을 병기하세요.

`votes` 테이블에는 `fetched_at` 컬럼이 없어 `data_freshness`의 `last_ingest_at`이 NULL로 나옵니다 — 이건 버그가 아니라 구조적 공백을 드러낸 것입니다.

## E. 검색의 한계

`search_bills`는 **부분문자열(ILIKE)** 검색입니다. tsvector 전문검색으로 바꾸지 마세요 — 한국어 형태소 분석기가 없어 recall이 오히려 나빠집니다(검토·기각됨).

- ⚠️ 질의 문자열이 **본문에 그대로 박혀야** 잡힙니다. 별칭은 0건입니다(`김영란법` → 0건). 단 통칭이 summary에 적혀 있으면 잡힙니다(`노란봉투법` → 1건). **통칭 → 정식명 치환이 호출하는 쪽의 몫입니다.**
- ⚠️ 동의어 갭도 있습니다(저출생 ↔ 저출산).
- ⚠️ **2글자 이하는 느립니다** — trigram이 3-gram이라 인덱스를 못 타고 seq scan으로 떨어집니다. 인덱스로 해결할 수 없는 구조적 한계입니다.
- ⚠️ 가운뎃점 표기가 섞여 있어 질의도 정규화가 필요할 수 있습니다.
- ⚠️ 광역 토픽은 `limit`을 크게(200+). 결과 수가 limit과 같으면 절단됐을 수 있습니다.

## F. 이 DB가 담지 않는 것

| 원하는 것 | 어디로 |
| --- | --- |
| 현행법·시행령 **본문**, 법령ID, **시행일자** | 법제처 (이 DB는 `prom_no`까지만 bridge) |
| 판례·헌재결정례·유권해석 | 법제처 |
| 법안 **전문(full text)** | 이 DB엔 `bill_name` + `summary`만 |
| **위원회 단계 표결** | 원천 미제공 (`votes`는 본회의만) |
| 위원회 **위원 명부**(누가 위원인지) | 이 DB에 없음 |
| 회의록·**발언**("누가 무엇을 말했나") | 범위 밖 (2026-06-28 제거 → websearch) |
| 청원·공청회·입법예고·여론 | 범위 밖 |

이 경계에 닿는 질문에는 **"이 DB 범위 밖"이라고 답하고** 외부 소스로 넘기는 게 맞습니다. 없는 데이터를 추정으로 채우지 마세요.
